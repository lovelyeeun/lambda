# cockpit 설계 결정 문서 — Work Item과 Chat의 분리

> 작성일: 2026-04-13
> 작성 배경: "한 채팅에서 여러 구매가 동시에 진행될 때 우측 패널 구성" 논의에서 출발
> 상태: **MVP 스코프 합의 완료 (2026-04-13)** — Phase 1을 스크립트 데모로 이번 프로토타입에 반영

---

## 1. 배경 — 우리가 풀어야 했던 질문

처음 던진 질문은 이거였다.

> "같은 채팅에서 상품을 여러 개 검색하거나, 구매요청 단계를 여러 개 동시에 진행할 경우, 우측 패널에 값이 어떻게 보여지는 게 이상적일까?"

이 질문을 풀려고 여러 모델을 검토하는 과정에서, 문제의 뿌리가 **"채팅과 작업(Work Item)의 관계를 어떻게 정의하느냐"**에 있다는 걸 발견했다.

## 2. 논의 과정에서 버린 선택지들

### 버린 선택지 A — 멀티탭(우측 패널 상단에 작업 탭 바)
초안으로 HTML 목업까지 만들었으나, 실제 cockpit-prototype의 우측 패널이 이미 **단일 파이프라인**(의도 분석 → 상품 검색 → 추천 → 장바구니 → 품의 → 결제 → 배송 → 완료) 구조로 구현돼 있어 탭을 얹기에는 구조 충돌이 컸다. 또한 한 화면에 탭을 강제로 만드는 방식은 사용자의 자연스러운 대화 흐름과 맞지 않는다는 판단.

### 버린 선택지 B — "한 채팅 = 한 작업" 강제 규칙
에이전트가 새 구매 의도를 감지하면 자동으로 채팅을 분기시키는 방식. 우측 패널 설계는 깔끔해지지만, Claude/Gemini처럼 자유로운 대화에 익숙한 사용자에게 **UI가 사용자를 강제하는 느낌**이 되어 어색하다. 또한 다음 케이스들을 풀 수 없다.
- 사용자가 다른 채팅에서 과거 작업 상태를 물어보는 경우
- 채팅 중간에 설정/리포트 등 다른 의도로 전환되는 경우

## 3. 채택한 개념 모델 — **Work Item ≠ Chat**

> **작업(Work Item)**은 독립적인 도메인 객체로 존재한다.
> **채팅(Chat)**은 그 작업들을 참조(reference)하는 대화 공간이다.

### 구체적 의미
- 구매 요청 건, 반품 건, 설정 변경 건 등은 모두 **고유 ID를 가진 Work Item**으로 저장된다 (예: `토너 재주문 #245`).
- 한 채팅에서 여러 Work Item이 생성/참조될 수 있다 (N:M 관계).
- 한 Work Item이 여러 채팅에서 참조될 수 있다.
- 채팅의 "제목"은 AI 자동 타이틀링일 뿐, 작업 경계가 아니다.

### 우측 패널의 재정의
우측 패널은 더 이상 "이 채팅의 구매 파이프라인"이 아니다. 새로운 정의:

> **우측 패널 = 지금 대화에서 관련된 도메인 객체(Work Item, Setting, Report 등)를 보여주는 컨텍스트 윈도우.**

객체 종류에 따라 모듈이 달라진다.
- Work Item (구매) → 기존 파이프라인 모듈
- Work Item (반품/CS) → 반품 진행 모듈
- Setting → 설정 미리보기 + 전용 페이지 딥링크
- Report/Data → 차트/데이터 모듈

## 4. 이 모델로 세 가지 핵심 케이스가 어떻게 풀리는가

### 케이스 1 — 한 채팅에서 여러 구매 동시 진행
청소기 검색 → 토너 재주문 → A4용지 비교를 한 채팅에서 다 한다.

- 에이전트가 각 구매를 Work Item `#245`, `#246`, `#247`로 생성
- 우측 패널 상단에 **"이 대화에서 언급된 Work Item 칩 스위처"** 표시 (예: `🔵 청소기 · 🟠 토너 · 🟣 A4용지`)
- 가장 최근 언급된 Work Item이 기본 활성 상태
- 이전 작업 칩 클릭 시 해당 Work Item의 파이프라인으로 전환
- Work Item이 1개뿐이면 스위처 숨김 → 기존 UX 유지

### 케이스 2 — 다른 채팅에서 "토너 주문 상태 어때?"
사용자가 새 채팅을 열고 과거 작업 상태를 묻는다.

- 에이전트가 사용자의 Work Item 목록에서 "토너" 매칭 → `#245` 탐색
- 우측 패널에 해당 Work Item의 **스냅샷 + "원래 채팅으로 이동" 딥링크** 표시
- 사용자가 이 채팅에서 이어가면 이 채팅도 `#245`를 참조 등록
- **결과**: Work Item은 어느 채팅에서 물어봐도 동일한 최신 상태를 보여줌

### 케이스 3 — 채팅 중 갑자기 설정 얘기
구매 얘기 중 "아 배송지 기본값 바꾸고 싶어".

- 설정은 별도 페이지의 AI 영역에서 처리하는 구조
- 우측 패널이 **"설정 미리보기 모듈 + 설정 페이지 딥링크"**로 모드 전환
- 또는 에이전트가 "설정 페이지로 이동할까요? 여기서 요약만 바꿀 수도 있어요"라고 제안
- 채팅은 그대로 유지됨, 우측 패널만 해당 의도에 맞춰 적응

## 5. 남은 디자인 난제 (후속 논의 필요)

1. **Work Item 식별 정확도**
   "토너 주문 어때?"가 `#245`인지 어제 한 `#241`인지 모호할 때. fallback UX 필요 (사용자 선택 / 최근 우선 / 상태 필터링 등).

2. **칩 스위처가 많아질 때의 UX**
   한 채팅에서 5개 이상 Work Item이 쌓이면 답답함. 상위 N개 + "더 보기" 혹은 자동 접기 필요.

3. **교차 채팅 동기화**
   채팅 A에서 만든 Work Item을 채팅 B에서 수정했을 때, 채팅 A의 컨텍스트에 어떻게 반영할지. 시스템 메시지 자동 삽입 방식 검토.

4. **간이 뷰 vs 전용 페이지 중복 문제**
   설정/리포트처럼 별도 페이지에 전용 뷰가 있는 경우, 우측 패널 간이 뷰의 정보량을 어디까지 보여줄지. 너무 많으면 중복, 너무 적으면 맥락 끊김.

5. **Work Item의 종류(type) 체계 정의**
   구매, 반품, CS, 설정, 리포트 등 어떤 타입을 first-class로 둘지. 타입별 모듈 스펙 정의 필요.

## 6. MVP 스코프 — 합의된 결정 (2026-04-13)

### 왜 이번 MVP에 포함하는가
1. **제품 정체성의 핵심** — "한 채팅에서 여러 구매가 동시에 진행된다"는 것은 ChatGPT/Gemini류 범용 챗봇과 cockpit을 구분짓는 결정적 차별점. 데모에서 이 동작을 보여주지 못하면 "구매 특화 AI"임을 1분 내에 각인시키기 어렵다.
2. **지금이 구조 전환 비용이 가장 싼 시점** — 아직 state가 많이 얽혀있지 않아 `workItems` 모델 도입이 쉽다. 나중에 미루면 리팩토링 비용이 몇 배가 된다.
3. **작업량 감당 가능** — 스크립트 데모로 한정하면 반나절~1.5일 범위.

### 포함 범위 (Phase 1 · 스크립트 데모)
- `WorkItem` 타입 정의 — `id`, `title`, `icon`, `color`, `status`, `cart`, `timelinePhase`, `approvalStep`, `searchRecords`, `sourcedProducts`, `candidateProducts` 등 파이프라인 snapshot
- 한 채팅 안에 `workItems: Record<id, WorkItem>` + `activeWorkItemId` 관리
- 하드코딩 키워드 트리거 기반 WI 자동 생성 (예: "청소기" / "토너" / "A4" 발화 시 신규 WI 생성)
- 우측 패널 헤더에 **Work Item 칩 스위처** — 2개 이상일 때만 노출, 1개일 땐 숨김 (케이스 1 권장대로 기존 UX 유지)
- "작업 현황" 섹션 분리 (구 "구매 컨텍스트")
  - **채팅 컨텍스트** (예산/배송지/결제수단/에이전트모드/최근주문) — WI 무관, 공유
  - **[선택된 Work Item]** (진행상황/검색기록/추출정보) — 활성 WI 기준

### 제외 (Phase 2~3으로 미룸)
- 실제 에이전트 의도 분석 기반 동적 WI 생성 (현재는 하드코딩 키워드 매칭)
- 교차 채팅 WI 참조/딥링크 (케이스 2)
- 비-구매 WI 모듈 — 반품/CS/설정/리포트 (케이스 3)
- WI 관리 전용 페이지 (C-10 구매요청서 관리를 확장하는 방향은 Phase 2에서 검토)

## 7. 데모 스크립트 (1~2분 분량)

1. 빈 상태에서 **"사무실 청소기 추천해줘"** → `#245 청소기` WI 생성, 칩 1개 (아직 숨김)
2. 추천 결과 → 장바구니 담기
3. 결제 전에 **"아 토너도 재주문해줘"** → `#246 토너` WI 생성 → **이때 처음 칩 스위처가 등장 (청소기 · 토너)**
4. 토너 WI에서 품의 → 승인 → 결제 → 완료 (`#246` 완료 상태)
5. **청소기 칩 클릭** → 장바구니 담긴 그대로 복귀 → 이어서 결제 진행
6. 둘 다 완료 → `/orders`에 두 건 모두 기록

**핵심 wow 포인트**: 3번 ("두 번째 구매 의도 발화 시 칩이 갑자기 생김") + 5번 ("멈춰있던 맥락이 그대로 복귀됨").

## 8. Work Item과 `/orders`의 관계

헷갈리기 쉬워서 명시:

| 레이어 | 정의 | 화면 |
|---|---|---|
| **Work Item** | 진행 중인 구매 **여정** (의도 → 검색 → 승인 → 결제 → 배송) | 채팅 우측 패널에만 존재 |
| **Order** | 결제 완료 후의 **주문 기록** | `/orders` (C-3 주문내역) |

**흐름**: Work Item 생성 → 파이프라인 진행 → 결제 완료 시점에 Order 레코드 생성 → Work Item 상태는 "완료"로 아카이브.

**MVP 처리**:
- 진행 중 Work Item은 오직 **채팅 내 우측 패널**에서만 가시화
- 완료된 것만 `/orders`에 기록
- 향후 필요하면 좌측 사이드바에 "진행 중 작업 (n)" 링크 또는 C-10 구매요청서 관리를 "Work Item 관리"로 확장 (Phase 2)

## 9. 남은 디자인 난제 (Phase 2~3에서 재논의)

(기존 §5에서 이동, 여전히 열려있는 질문들)

1. **Work Item 식별 정확도** — 하드코딩 키워드 매칭을 넘어 에이전트 의도 분석으로 전환 시 모호성 해소 방법
2. **칩 스위처가 많아질 때의 UX** — 5개 이상 쌓이면 상위 N + "더 보기" 필요
3. **교차 채팅 동기화** — 채팅 A의 WI를 채팅 B에서 수정 시 반영 방식
4. **간이 뷰 vs 전용 페이지 중복 문제** — 우측 패널 모듈과 전용 페이지 정보량 균형
5. **Work Item type 체계 정의** — 구매/반품/CS/설정/리포트 중 first-class 대상
