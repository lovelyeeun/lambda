## 1. 에어서플라이의 기존 가치와 저항 분석

Lambda은 새로운 제품이지만, 에어서플라이가 이미 증명한 가치 위에 세워진다. 기존 가치를 이해해야 Lambda이 뭘 계승하고 뭘 새로 만들어야 하는지가 명확해진다.

### 에어서플라이의 증명된 가치

| 가치 | 설명 | 누가 체감하나 |
|------|------|-------------|
| **구매처 통합** | 외부몰 상품 링크를 복붙하면, 구매처가 여러 개여도 계정 하나로 구매 가능 | 관리자 + 구성원 |
| **법인카드 세부내역 가시화** | 법인카드로 결제 시 가려지는 세부 구매내역이 항목별로 보임 | 관리자 (재무/경영지원) |
| **보안 문제 해결** | 개인 쇼핑몰 계정 공유 없이 기업 구매 가능 | 관리자 (IT/보안) |
| **정산 시간 90%+ 단축** | 월말 정산 시 영수증 취합/대조 자동화 | 관리자 (재무) |
| **지점 관리 용이** | 다수 지점의 구매를 본사에서 통합 관리, 지점별 매출 카운트 | 관리자 (본사) |
| **스펙 기반 정확한 구매** | 구성원이 원하는 정확한 스펙의 상품을 찾아 바로 구매요청 | 구성원 |

### 기존 저항 포인트

```
[가입] ──── ⚡ 거대한 골짜기 ⚡ ──── [세팅 완료] ─── 1개월 ─── [정산 완료] ── 💡 와우!

         상품등록, 결제수단 설정                    구매 축적           "아 이건 진짜 편하다"
         팀원초대, 가입시키기                                        세부내역 정리
         권한설정 (무지 복잡)                                       계정 통합 정산
         "이걸 언제 다 해..."                                      "이건 못 떠나겠다"
```

| 저항 | 원인 | 발생 시점 |
|------|------|---------|
| "수수료가 비싸" | 구매 건당 비용이 직접 보임 | 구매 시점마다 |
| "구매내역 언제 가져와?" | 정산 전까지 데이터가 안 보임 | 구매 직후 |
| "쿠팡 와우배송 안 돼?" | 특정 몰 기능/배송 제약 | 상품 탐색 시 |
| "세팅 언제 다 해?" | 수동 설정 항목이 너무 많음 | 가입 직후 |
| "권한 설정 복잡함" | 메뉴 기반 UI + 개념 이해 필요 | 온보딩 중 |

### 핵심 구조적 문제

**에어서플라이의 가치는 실재하지만, 그 가치에 도달하는 시간이 너무 길다.**
- 세팅 완료까지: 반나절~1일 (관리자의 노동)
- 진짜 와우(정산 체감)까지: 최소 1개월
- 이 기간 동안 보이는 건 수수료뿐 → 이탈

## 2. Lambda이 해결하는 것

### 핵심 명제

> **에어서플라이의 기존 가치(구매 통합, 정산 자동화, 지점 관리)는 그대로 계승하되,
> 그 가치에 도달하는 경로를 AI가 극적으로 단축한다.**

```
As-Is (에어서플라이):
  [가입] ──── 수동 세팅 (반나절) ──── 1개월 사용 ──── [💡 정산 와우]

To-Be (Lambda):
  [가입] ── AI 진단 세팅 (15분) ── [💡 첫 탐색 와우] ── [💡 실시간 축적 와우] ── [💡 정산 와우]
```

두 가지를 동시에 한다:
1. **기존 가치의 도달 경로 단축** — 온보딩을 AI가 대화로 처리
2. **새로운 가치 추가** — "말하면 사지는" 대화형 구매 경험

## 3. AI 제품들의 MVP 설계 패턴 분석

### 공통 패턴: "하나의 문장으로 설명 가능한 핵심 경험"

| 제품 | MVP 핵심 경험 (한 문장) | 첫 출시에서 뺀 것 | 팀 규모 | 출시까지 |
|------|----------------------|-----------------|---------|---------|
| **ChatGPT** | "AI와 대화하기" | 플러그인, API, 파인튜닝, 이미지 — 전부 뺌 | 제품팀 ~10-15명 | 모델 위에 ~3-4개월 |
| **Claude** | "안전하고 유능한 AI 대화" | 엔터프라이즈 기능, 커스텀 KB, Projects | 초기 ~20명 | 창업 후 ~18개월 |
| **Cursor** | "기존 에디터에서 AI 코드 제안" | 멀티파일 리팩토링, 팀 협업, 엔터프라이즈 | **2-3명** | YC S22 배치 중 |
| **NotebookLM** | "내 문서에 질문하기" | 실시간 협업, 퍼블리싱, 고급 분석 | Google Labs 소규모 팀 | 기존 연구 활용 |
| **Manus** | "AI에게 작업 위임하기" | 범용성 → 특정 버티컬만 집중 후 확장 | 소규모 스타트업 | 빠른 베타 루프 |
| **Lilys.ai** | "영상/문서를 AI가 정리" | 다국어, 고급 편집, 팀 기능 | ~5-10명 추정 | 한국어 특화 빠른 출시 |

### Lambda에 적용할 핵심 인사이트

**① "모델은 남의 것, UX는 내 것" 전략**
- 모든 성공 제품이 모델 학습이 아닌 UX/인터페이스에 집중
- Lambda: LLM API 활용 + 에어서플라이 상품DB/결제 인프라/구매 프로세스 지식이 진짜 자산

**② MVP에서 "기능"이 아니라 "경험"을 정의**
- ChatGPT: "자연어 처리 기능" X → "대화 경험" O
- Cursor: "코드 생성 기능" X → "Tab 하나로 코드 완성 경험" O
- Lambda: **"말하면 회사 물건이 알아서 사지는 경험"**

**③ "Research Preview" 포지셔닝**
- ChatGPT의 "연구 프리뷰" → 기대치 관리 + 피드백 수집
- Lambda: "AI 구매 비서 얼리 액세스" → 고객이 함께 만드는 느낌

## 4. 10명 이하 팀의 MVP 규모 설정

### 가이드라인: "4주 안에 첫 유저가 쓸 수 있는가?"

| 원칙 | 적용 |
|------|------|
| **하나의 Job-to-be-Done** | "필요한 물건을 대화로 찾아서 구매 요청까지" — 이것만 완벽하게 |
| **하나의 데이터 소스** | 에어서플라이 상품DB만. 외부 커넥터는 아키텍처만 설계 |
| **하나의 사용자 유형 우선** | 구매 요청자(구성원)의 경험을 먼저 완성. 관리자 뷰는 최소한으로 |
| **하나의 플랫폼** | 웹 먼저. 모바일은 뒤로 |

### MVP 스코프 (Phase 1 = 4-6주)

```
┌─── MVP에 포함 ───────────────────────────────────────┐
│                                                        │
│  ✅ AI 진단 기반 온보딩 (유형 분류 → 자동 세팅 추천)      │
│  ✅ 대화형 상품 탐색 + 최저가 비교 + 옵션 분석             │
│  ✅ 상품 카드 인라인 표시 (판단 근거 접이식)               │
│  ✅ "이걸로 해줘" → 구매요청서 자동 생성                   │
│  ✅ 기본 승인 플로우 (관리자에게 알림 → 승인/반려)          │
│  ✅ 결제 실행 (에어서플라이 기존 결제 인프라 활용)           │
│  ✅ 구매 즉시 실시간 지출 기록 노출 (정산 와우 앞당기기)     │
│                                                        │
└────────────────────────────────────────────────────────┘

┌─── MVP에서 제외 (설계만) ─────────────────────────────────┐
│                                                           │
│  📐 커넥터 아키텍처 (인터페이스 스펙 + L1/L2/L3 정의)       │
│  📐 @서비스명 / 슬래시 커맨드 입력 파싱 구조                │
│  📐 관리자 대시보드 UI 구조                                │
│  📐 유형 자동 재분류 로직 설계                              │
│                                                           │
└───────────────────────────────────────────────────────────┘

┌─── MVP에서 완전 제외 ─────────────────────────────────────┐
│                                                           │
│  ❌ 외부 공급사 커넥터 구현                                 │
│  ❌ 지출 분석/구매 건강도 대시보드                           │
│  ❌ 모바일 앱                                              │
│  ❌ 다중 결제수단 관리                                      │
│  ❌ 기업 간 벤치마킹                                        │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

### 왜 이 범위인가

Cursor가 2-3명으로 MVP를 만든 비결은 "VS Code라는 기존 껍데기 위에 AI 하나만 얹기". Lambda도 동일 전략:
- **기존 껍데기** = 에어서플라이의 상품DB + 결제 인프라 + 배송 추적
- **새로 만드는 것** = 대화형 인터페이스 + AI 에이전트 + 승인 플로우 + AI 온보딩

## 5. 기업 유형 분류 시스템 및 진화 구조

### 현재 분류 체계 (6가지)

에어서플라이 기존 고객 데이터 기반으로 이미 검증된 클러스터링:

| 유형 | 기업수 | 매출비중 | 임직원(중앙) | 월평균거래액(중앙) | 월평균주문수(중앙) | 주문당단가 | 카테고리 다양성 | 대표 카테고리 TOP3 |
|------|-------|---------|-----------|----------------|---------------|----------|-------------|----------------|
| **A. 공간운영형** | 11 | 22.3% | 43 | ₩7,567,363 | 66 | ₩80,000 | 11 | 생활/건강 43.6%, 식품 25.0%, 가구/인테리어 9.8% |
| **B. F&B/식음업형** | 13 | 11.8% | 43 | ₩4,274,650 | 48 | ₩60,000 | 12 | 생활/건강 43.9%, 식품 27.0%, 문구/사무용품 13.3% |
| **C. 제조/하드웨어형** | 24 | 19.3% | 49 | ₩4,050,121 | 18.5 | ₩140,000 | 9 | 공구 33.6%, 생활/건강 17.3%, 디지털/가전 16.0% |
| **D. IT 오피스형** | 28 | 12.3% | 58 | ₩2,050,783 | 13.5 | ₩50,000 | 8.5 | 식품 30.1%, 디지털/가전 27.2%, 문구/사무용품 17.0% |
| **E. 이커머스/유통형** | 20 | 23.6% | 124 | ₩5,260,636 | 46.5 | ₩80,000 | 11 | 생활/건강 26.7%, 식품 21.8%, 문구/사무용품 16.7% |
| **F. 전문서비스형** | 23 | 10.7% | 34 | ₩1,657,097 | 9 | ₩70,000 | 7 | 생활/건강 24.5%, 식품 22.9%, 디지털/가전 12.6% |

### 유형 시스템이 Lambda에서 하는 역할

**① 온보딩 시: 초기 세팅 자동화의 근거**
- 신규 기업이 업종/규모/주요 구매품목을 입력하면 → 가장 유사한 유형으로 진단
- 해당 유형의 평균 데이터를 기반으로 구매 관리 뷰, 권한 체계, 예산 기준선을 추천

**② 운영 시: "비슷한 기업은 이렇게 씁니다" 맥락 제공**
- "같은 IT 오피스형 28개사의 평균 주문당 단가는 ₩50,000인데, 이 상품은 그 안에 있어요"
- Day 1부터 비교 맥락을 제공할 수 있는 근거 (콜드스타트 해결)

**③ 분석 시: 유형별 벤치마크 기준선**
- "귀사는 IT 오피스형 평균 대비 디지털/가전 지출이 40% 높아요"
- 절감 포인트 제안의 근거

### 유형 진화 구조 (2-레이어)

**레이어 1: 전체 클러스터 재정의 (주기적)**

```
기업 수 증가 → 새 패턴 출현 → 주기적 재클러스터링

현재: 6개 유형 (119개사 기반)
→ 300개사: "병원/의료형", "교육기관형" 등 분화 가능
→ 500개사+: 더 세분화 또는 유형 내 서브클러스터
```

- 클러스터링 주기: 분기 1회 또는 기업 수 임계치 도달 시
- 방식: 기존 변수(거래액, 주문수, 단가, 카테고리 비중) 기반 k-means 또는 유사 기법
- 유형 변경 시 기존 기업에 "추천 세팅 업데이트" 알림

**레이어 2: 개별 기업의 유형 진화 (상시)**

```
[가입]                    [3개월 후]                   [6개월 후]
D. IT 오피스형            구매 패턴 변화 감지             유형 재진단
추천 세팅 적용       →    "장비 구매 비중이 늘었어요"  →   C. 제조/하드웨어형에 가까움
                         "뷰 추가할까요?"               "권한 체계 조정 추천"
```

- 개별 기업의 실제 구매 데이터가 쌓이면, 가입 시 진단과 달라질 수 있음
- AI가 패턴 변화를 감지하고 "세팅을 조정할까요?"라고 제안
- 세팅이 한 번으로 끝나는 게 아니라 **계속 최적화되는 경험**

## 6. 온보딩 플로우 상세 설계

### 설계 원칙

> **"빈 캔버스에서 시작하지 않는다."**
> 기존 119개사의 구매 데이터가 있으니, 신규 기업은 그 위에서 미세 조정만 하면 된다.
> "세팅 언제 다 해..." → "확인 몇 번이면 끝"

### 온보딩 전체 맵

```
Step 1: 기업 기본정보 입력 (2분)
  ↓
Step 2: AI 유형 진단 (10초)
  ↓
Step 3: 세팅 추천 확인 (3분)
  ├── 구매 관리 뷰
  ├── 권한 체계
  └── 예산 기준선
  ↓
Step 4: 결제수단 연결 (2분)
  ↓
Step 5: 팀원 초대 (2분)
  ↓
Step 6: 첫 구매 체험 (1분)
  ↓
🎯 온보딩 완료 (약 10-15분)
```

### Step 1: 기업 기본정보 입력

대화형으로 진행. 폼이 아니라 채팅.

```
🤖 "안녕하세요! Lambda 시작을 도와드릴게요.
    몇 가지만 알려주시면 맞춤 세팅을 해드립니다."

🤖 "회사명이 어떻게 되세요?"
👤 "위드테크"

🤖 "업종은요?"
👤 "소프트웨어 개발"

🤖 "직원 수는 대략 몇 명이에요?"
👤 "48명"

🤖 "주로 어떤 물품을 구매하세요? 편하게 말씀해주세요."
👤 "간식이랑 음료, 모니터 같은 장비, 문구류 좀"
```

**수집하는 정보:** 회사명, 업종, 직원 수, 주요 구매 카테고리
**기존 대비 개선:** 폼 10개 필드 → 자연어 대화 4턴 (체감 부담 대폭 감소)

### Step 2: AI 유형 진단

입력된 정보를 기존 6개 유형 데이터와 매칭.

```
🤖 "비슷한 규모의 IT 기업들 구매 패턴을 분석해봤어요."

   ┌─────────────────────────────────────────────┐
   │  📊 진단 결과: D. IT 오피스형                  │
   │                                              │
   │  유사 기업 28개사의 평균:                       │
   │  • 월 주문 약 13-14건                         │
   │  • 주문당 약 ₩50,000                         │
   │  • 식품 30% · 디지털/가전 27% · 문구 17%       │
   │                                              │
   │  "위드테크와 비슷한 기업들이                     │
   │   Lambda을 이렇게 쓰고 있어요."               │
   └─────────────────────────────────────────────┘
```

**핵심:** "28개사가 이미 이렇게 쓰고 있다"는 사회적 증거(social proof)가 세팅 신뢰감을 줌.

### Step 3: 세팅 추천 확인

유형별로 미리 정의된 추천 템플릿을 제안. 관리자는 확인/수정만.

```
🤖 "IT 오피스형 기업들이 가장 많이 쓰는 구조로 세팅해볼게요."

   📌 추천 구매 관리 뷰:
   ├── 🍎 간식/음료 (식품)
   ├── 🖥️ 장비/가전 (디지털/가전)
   ├── ✏️ 사무용품 (문구/사무용품)
   └── 📦 기타

   [이대로 할게요]  [뷰 추가/수정]
```

```
🤖 "권한은 이렇게 추천드려요."

   👑 관리자 (경영지원/재무)
      → 전체 승인 + 정산 조회 + 예산 설정

   ✅ 팀장급
      → ₩100,000 이하 자동승인 (초과 시 관리자 승인)

   📝 일반 구성원
      → 구매요청만 가능

   "48명 규모면 보통 관리자 2-3명, 팀장급 5-8명 정도예요."

   [이대로 할게요]  [금액/권한 수정]
```

```
🤖 "월 예산 기준선도 잡아드릴까요?"

   IT 오피스형 평균 월 거래액: ₩2,050,783
   48명 기준 1인당 약 ₩42,700/월

   "처음엔 월 ₩2,500,000로 시작하는 건 어때요?
    넘으면 알려드릴게요."

   [좋아요]  [다른 금액으로]
```

**기존 대비 개선:**
- 에어서플라이: 구매 그룹을 뭘로 만들지 고민 → 메뉴 찾기 → 수동 생성 → 권한 하나씩 설정
- Lambda: AI가 유형 기반으로 추천 → "이대로 할게요" 3번 클릭이면 완료

### Step 4: 결제수단 연결

```
🤖 "결제수단을 연결할게요. 법인카드를 쓰시나요?"
👤 "응"

🤖 "카드 등록은 보안상 직접 입력해주세요."
   [카드 등록 화면으로 이동]

   ✅ 등록 완료!

🤖 "참고로 Lambda을 통한 구매는 모든 세부내역이 자동 기록돼요.
    법인카드 명세서에 '에어서플라이'로만 뜨더라도,
    안에서 뭘 샀는지 항목별로 다 볼 수 있어요."
```

**핵심:** 기존 에어서플라이의 핵심 가치(법인카드 세부내역 가시화)를 이 시점에 미리 언급해서, 정산 와우를 "예고"함.

### Step 5: 팀원 초대

```
🤖 "팀원을 초대할게요. 어떻게 하면 편하세요?"

   [슬랙 워크스페이스 연동]  [이메일로 초대]  [CSV 업로드]

   (슬랙 연동 선택 시)

🤖 "48명 확인됐어요. 부서 정보도 가져왔습니다."

   마케팅팀 12명 → 📝 일반 구성원
   개발팀 18명 → 📝 일반 구성원
   디자인팀 8명 → 📝 일반 구성원
   경영지원팀 6명 → 👑 관리자 2명 + 📝 일반 4명
   경영진 4명 → ✅ 팀장급

   "이렇게 배치하면 될까요?"

   [이대로 할게요]  [개별 수정]
```

**기존 대비 개선:**
- 에어서플라이: 팀원 이메일 하나씩 입력 → 가입 기다리기 → 권한 하나씩 설정
- Lambda: 슬랙 연동 한 번 → 부서 자동 인식 → 권한 자동 추천 → 확인

### Step 6: 첫 구매 체험 (= Day 1 와우 모먼트)

온보딩의 마지막을 "기능 설명"이 아니라 **실제 경험**으로 마무리.

```
🤖 "세팅 완료! 🎉 바로 써볼까요?
    지금 필요한 물건이 있으면 말해보세요."

👤 "사무실 복합기 토너 하나 필요해"

   ⏱️ 3초

🤖 "2개 찾았어요."

   ┌───────────────────────────────────────────┐
   │  🏷️ 삼성 CLT-K510S        ₩52,800  ⭐ 추천 │
   │  📦 내일 도착 | ⭐ 4.8 | 반품률 0.1%        │
   │                                           │
   │  💡 IT 오피스형 기업들이 가장 많이 사는 제품    │
   │     카테고리 평균가 대비 12% 저렴              │
   │                                           │
   │  [구매요청]  [더 보기]                       │
   └───────────────────────────────────────────┘

👤 "이걸로 해줘"

🤖 "구매요청서 작성했어요."
   부서: 경영지원팀 | 용도: 사무실 소모품
   → 김미영 관리자에게 승인요청을 보냈습니다.

   📱 김미영님에게 슬랙 알림이 갔어요.
   승인되면 바로 알려드릴게요!
```

**이 순간의 와우 포인트:**
- 세팅이 끝나자마자 바로 작동한다
- 말했더니 3초 만에 결과가 나온다
- 구매요청서, 승인 라우팅이 자동이다
- "IT 오피스형 기업들이 가장 많이 사는 제품" — 회사 이력은 없지만 유형 데이터로 맥락 제공

### 기존 에어서플라이 고객 전환 시 특별 플로우

기존 고객은 이미 구매 이력이 있으므로, 콜드스타트 문제가 없다.

```
🤖 "기존 에어서플라이 계정이 있으시네요!
    지난 6개월 구매 데이터를 가져올까요?"

   [가져오기]

🤖 "분석 완료! 총 142건, ₩24.3M 구매하셨어요."

   📊 자동 분석 결과:
   • 유형: E. 이커머스/유통형
   • 최다 구매 카테고리: 생활/건강 (32%), 식품 (24%)
   • 평균 주문당 단가: ₩85,000
   • 재주문 비율: 34% (이 부분을 자동화하면 시간 절약 가능!)

   "기존 폴더 구조와 권한도 그대로 가져왔어요.
    Lambda에 맞게 약간 조정할 부분을 추천해드릴게요."
```

**핵심:** 기존 고객은 Day 1부터 "우리 회사 과거 구매 대비" 비교가 가능 → 와우 모먼트가 즉시 발동.

### 6-A. Progressive Onboarding: 타이밍 전략

> **v3 핵심 전환: "온보딩 = 관문"에서 "온보딩 = 레이어"로**

#### 왜 기존 설계(Step 1~6 순차 진행)를 바꾸는가

기존 v2 설계는 가입 → AI 진단 → 세팅 확인 → 결제 → 팀원초대 → 첫 구매까지 **순차적 10-15분**을 요구했다. 이건 기존 에어서플라이의 "세팅 반나절"보다는 낫지만, 근본적인 문제가 있다:

1. **첫 채팅이 반드시 "나는 무슨 회사야"로 시작하지 않는다.** 사용자의 첫 심리는 "이게 뭔지 한번 써보자"이지 "회사 정보를 입력하자"가 아니다.
2. **Lilys.ai 같은 폼 방식은 Lambda에 안 맞는다.** Lilys는 역할/관심사 2-3개만 고르면 피드가 채워지는 콘텐츠 소비 도구. Lambda은 업종·규모·구매품목·결제수단·팀구조·승인체계까지 필요 — 폼으로 한번에 받으면 기존 저항이 반복된다.
3. **가치 체험 전에 세팅을 요구하면 이탈한다.** ChatGPT가 가입 즉시 대화가 가능했듯, Lambda도 가입 즉시 상품 탐색이 가능해야 한다.

#### Progressive Onboarding 구조

```
온보딩을 "관문"이 아니라 "레이어"로 만든다.
가치를 먼저 체험하게 하고, 더 깊은 가치가 필요한 순간에 다음 세팅을 요청한다.

┌─────────────────────────────────────────────────────────┐
│  Layer 0: 가입만 완료                                     │
│  → 상품 탐색/비교 바로 가능 (가치 체험)                     │
│  → 회사 정보 없이도 시장 데이터 기반 추천                    │
│                                                          │
│  Layer 1: 회사 기본정보 입력 (AI가 대화 중 자연스럽게 수집)   │
│  → 유형 진단 + 개인화 추천 활성화                           │
│  → "비슷한 회사들이 이렇게 쓰고 있어요" 맥락 제공            │
│                                                          │
│  Layer 2: 결제수단 연결                                    │
│  → 실제 구매요청/결제 가능 (핵심 전환)                      │
│  → 트리거: "이걸로 해줘" 클릭 시                            │
│                                                          │
│  Layer 3: 팀원 초대 + 권한 + 구매 관리 뷰 설정               │
│  → 조직 구매 관리 활성화 (Lock-in)                         │
│  → 트리거: "팀원도 쓸 수 있어?" 또는 관리자 기능 필요 시      │
│                                                          │
│  Layer 4: 예산 설정 + 승인 플로우                           │
│  → 거버넌스 활성화                                        │
│  → 트리거: 구매 건수가 쌓이거나 관리자가 통제 필요 시         │
└─────────────────────────────────────────────────────────┘
```

#### 진입점 시나리오

**① "바로 써볼게요" — 탐색형 진입 (예상 다수)**

```
[가입 완료 → 빈 채팅 화면]

🤖 "안녕하세요! 회사에서 필요한 물건,
    저한테 말만 하시면 찾아드릴게요."

   [가이드 투어]  [바로 써볼게요]

👤 "사무실 모니터 하나 찾아줘"                    ← Layer 0

🤖 "27인치 모니터 3개를 비교해봤어요."
   [상품 카드 표시]

   💡 "어떤 회사에서 쓰시는지 알려주시면
      비슷한 회사들이 많이 사는 모델을 추천해드릴 수 있어요."

👤 "소프트웨어 개발회사인데 50명 정도"              ← Layer 1 자연 전환

🤖 "IT 오피스형 기업 28개사 데이터 기반으로 다시 찾았어요.
    이 모니터가 비슷한 회사에서 가장 많이 팔렸고,
    카테고리 평균 대비 12% 저렴해요."

👤 "이걸로 해줘"

🤖 "구매하려면 결제수단을 연결해야 해요.            ← Layer 2 자연 전환
    법인카드 등록하시면 바로 주문 넣어드릴게요."

   [카드 등록하기]  [나중에]
```

핵심: **사용자의 행동이 트리거**다. AI가 "다음 단계를 해야 합니다"라고 끊지 않고, 사용자가 더 깊은 가치를 원하는 순간에 다음 레이어가 자연스럽게 열린다.

**② "가이드 투어" — 적극적 진입 (관리자 다수)**

```
🤖 "좋아요! 5분이면 기본 세팅을 끝낼 수 있어요.
    몇 가지만 여쭤볼게요."

    → 기존 v2 Step 1-3 대화형 온보딩 진행
    → 단, 각 단계마다 [나중에 하기] 옵션 제공
    → 결제수단/팀원초대는 스킵 가능
```

**③ "원래 에어서플라이 쓰고 있었어" — 전환 진입**

```
🤖 "기존 에어서플라이 계정이 있으시네요!
    지난 6개월 구매 데이터를 분석해서 맞춤 세팅을 해드릴까요?"

    [데이터 가져오기]

    → 기존 폴더/권한/결제수단 마이그레이션
    → Layer 1~4 대부분 자동 완료
    → "개선 포인트만 확인해주세요" 모드
```

#### Layer 전환 트리거 맵

| 전환 | 트리거 (사용자 행동) | AI 행동 | 필수 vs 선택 |
|------|-------------------|---------|-------------|
| 0→1 | 상품 탐색 중 회사 맥락 필요 시 | "어떤 회사에서 쓰시는지 알려주시면..." | 선택 (없어도 탐색 가능) |
| 0→1 | 가이드 투어 선택 | 대화형 정보 수집 시작 | 능동적 선택 |
| 1→2 | "구매해줘" / "이걸로 해줘" | "결제수단을 연결해야 해요" | 구매 시 필수 |
| 1→3 | "팀원도 쓸 수 있어?" / 팀원 언급 | "팀원 초대하고 구매 관리 기준을 잡아드릴까요?" | 선택 |
| 2→4 | 구매 5건 이상 또는 관리자가 승인 설정 요청 | "승인 규칙을 정해볼까요?" | 선택 |
| any | 관리자가 직접 "전체 세팅 하고 싶어" | 가이드 투어 모드로 전환 | 능동적 선택 |

#### v2 대비 변경 요약

| 항목 | v2 (기존) | v3 (Progressive) |
|------|----------|------------------|
| 온보딩 타이밍 | 가입 직후 순차 진행 | 사용자 행동 기반 점진적 전개 |
| 최초 진입 경험 | AI 진단 질문으로 시작 | 빈 채팅 + 바로 탐색 가능 |
| 정보 수집 방식 | 대화형이지만 한번에 수집 | 필요한 순간에 필요한 만큼 |
| 결제수단 등록 | 온보딩 Step 4에서 필수 | 실제 구매 시도 시 트리거 |
| 팀원 초대 | 온보딩 Step 5에서 권장 | 팀 기능 필요 시 트리거 |
| 이탈 리스크 | 세팅 중간 이탈 가능 | 각 레이어가 독립 — 어디서든 멈춰도 OK |

> **주의: v2의 Step 1~6 대화 스크립트는 삭제하지 않는다.** "가이드 투어"를 선택한 사용자에게 그대로 제공되며, Layer 3-4 전환 시에도 활용된다. Progressive Onboarding은 Step들의 "순서와 필수성"을 바꾸는 것이지, 각 Step의 내용을 바꾸는 게 아니다.

### 6-B. 구매 관리 패턴 실데이터 분석 & 초기 뷰 세트 템플릿

> 에어서플라이 운영 DB (department_entity) 205개 기업, 폴더(구 에어서플라이 용어) 3개 이상 보유 기업 기준 분석
> **Lambda에서는 "폴더"를 "뷰(Saved View)"로 재정의한다.** 사용자에게는 폴더처럼 보이지만, 내부는 상품 컨텍스트 기반 필터. 하나의 상품이 여러 뷰에 동시 소속 가능.

#### 발견된 4가지 관리 패턴

| 패턴 | 기업 수 | 비율 | 평균 뷰 수 | 특징 |
|------|---------|------|------------|------|
| **부서/팀 중심** | 97개 | 47% | 8개 | 조직도와 1:1 매핑. "경영지원팀, 마케팅팀, 개발팀" |
| **용도/프로젝트/브랜드별** | 77개 | 38% | 8개 | 구매 목적 기반. "소모품, IT장비, 행사, 브랜드명" |
| **혼합형 (지점+부서)** | 16개 | 8% | 14개 | 본사 부서 + 지점 뷰 공존 |
| **지점/매장 중심** | 15개 | 7% | 19개 | 지점명이 곧 뷰 이름. 가장 많은 뷰 수 |

> **핵심 전환:** 에어서플라이에서 이 패턴들은 "폴더"로 구현됐지만, Lambda에서는 **상품 컨텍스트의 필터 조건이 다른 것**일 뿐이다. 부서형 = `context.team` 기준 뷰, 용도형 = `context.purpose` 기준 뷰, 지점형 = `context.location` 기준 뷰. 상품이 여러 맥락에 동시 소속 가능하므로 폴더의 한계(A4용지가 사무용품이면서 월간구매이면서 총무팀 관할)가 해결된다.

```

              백엔드팀, 프론트엔드팀, UI/UX팀, QA팀, 마케팅팀...
아이리스브라이트 (26개): 피플팀, 콘텐츠기획1~3팀, 제품개발팀, DevOps, MD팀...
코드잇 (15개): Product, Content, Brand Experience, Finance, People&Culture...
```

**용도형 (38%) — 기존 6개 유형으로 예측하기 어려운 패턴**

```
버킷플레이스/오늘의집 (18개): 소모품(WOS), IT소모품, 고객경품, 물류센터,

베이크플러스 (19개): VALRHONA SELECTION, ELLE & VIRE, R&D, RETAIL...
```

**지점형 (7%) — 공간운영/F&B/프랜차이즈 기업**

```
스파크플러스 (49개): 고객그룹_역삼점, 고객그룹_강남점... + 신규지점(초도물품)
패스트파이브 (48개): 선릉1호점, 신논현2호점, 역삼2호점, 여의도점...
```

```
새롬종합관리 (31개): 본사 + 아파트 단지명 (이편한세상, 래미안, 자이...)
```

#### 온보딩 시 구매 관리 기준 선택 플로우

v2에서는 유형 진단 후 폴더를 자동 추천했지만, 실데이터를 보면 **같은 유형 내에서도 관리 패턴이 다르다** (IT 오피스형인데 부서형일 수도, 용도형일 수도 있음). 따라서 관리 기준 자체를 사용자에게 선택하게 하고, 그에 맞는 **초기 뷰 세트**를 자동 생성하는 것이 더 정확하다.

```
🤖 "구매를 어떤 기준으로 관리할까요?
    비슷한 회사들은 이렇게 쓰고 있어요."

   🏢 부서별 — 가장 많이 쓰는 방식 (47%)
      "팀/부서마다 구매를 따로 관리하고 싶을 때"
      → 경영지원, 마케팅, 개발, 디자인...
   
   📍 지점별 — 다지점 기업에 최적
      "지점/매장마다 구매를 분리하고 싶을 때"
      → 강남점, 역삼점, 판교점...
   
   🎯 용도별 — 두 번째로 많이 쓰는 방식 (38%)
      "구매 목적에 따라 분류하고 싶을 때"
      → 사무용품, IT장비, 간식/음료, 행사...

   [부서별]  [지점별]  [용도별]  [직접 만들기]
```

> **"직접 만들기"를 선택하면?** 빈 상태에서 시작하지 않는다. 사용자가 직접 뷰 이름을 입력하면 AI가 상품 컨텍스트 필터 조건을 자동 매핑한다. 예: "마케팅 행사" → `context.team = 마케팅 AND context.purpose = 행사`.
> 
> **뷰와 폴더의 결정적 차이:** 뷰는 필터 조건이므로, 하나의 상품이 "마케팅팀" 뷰와 "행사용품" 뷰에 동시에 나타날 수 있다. 기존 폴더에서는 불가능했던 구조.

#### 기업 유형 × 관리 패턴: 초기 뷰 세트 템플릿

선택된 관리 기준 + AI 진단 유형을 조합해서 구체적인 초기 뷰 세트를 추천한다. 각 뷰는 상품 컨텍스트의 특정 필드를 필터 조건으로 가진다.

**A. 공간운영형 + 지점별**
```
초기 뷰 세트 (필터: context.location)
├── 📌 본사 (관리자 스코프)
├── 📌 [지점명 1]
├── 📌 [지점명 2]
├── 📌 [지점명 N]
└── 📌 신규지점/초도물품
예산 정책: 지점당 월 ₩700K (중앙값 기반)
승인: 지점장 ₩200K 이하 자동 → 초과 시 본사 승인
추천 근거: 스파크플러스(49개), 패스트파이브(48개) 등 운영 패턴
```

**B. F&B/식음업형 + 지점별**
```
초기 뷰 세트 (필터: context.location)
├── 📌 오피스/본사 (관리자 스코프)
├── 📌 [매장명 1]
├── 📌 [매장명 2]
└── 📌 R&D/신메뉴 개발 (필터: context.purpose)
예산 정책: 매장당 월 ₩400K
승인: 매장장 ₩100K 이하 자동
추천 근거: 도레도레(39개), HAGOHAUS(27개), 에이블커피그룹(16개)
```

**C. 제조/하드웨어형 + 부서별**
```
초기 뷰 세트 (필터: context.team)
├── 📌 구매팀 (관리자 스코프)
├── 📌 생산/제조
├── 📌 연구개발
├── 📌 품질관리
├── 📌 경영지원
└── 📌 현장/물류
예산 정책: 월 ₩4M (중앙값), 주문당 ₩140K
승인: 팀장 ₩300K 이하 자동 → 초과 시 구매팀 승인
주의: 공구/장비 단가가 높아 승인 기준선을 다른 유형보다 높게
```

**D. IT 오피스형 + 부서별**
```
초기 뷰 세트 (필터: context.team)
├── 📌 경영지원 (관리자 스코프)
├── 📌 개발
├── 📌 디자인
├── 📌 마케팅
├── 📌 기획/PM
└── 📌 공용/간식 (필터: context.purpose = 복리후생)
예산 정책: 월 ₩2M (중앙값), 주문당 ₩50K
승인: 팀장 ₩100K 이하 자동
추천 근거: IT 오피스형 28개사 평균
```

**D. IT 오피스형 + 용도별 (대안)**
```
초기 뷰 세트 (필터: context.purpose)
├── 📌 사무용품/소모품
├── 📌 IT장비/가전
├── 📌 간식/음료
├── 📌 도서/교육
└── 📌 기타
예산 정책: 동일
승인: 카테고리별 차등 (IT장비 ₩200K, 간식 ₩50K)
추천 근거: 버킷플레이스, 코드잇 등 용도형 선택 기업
```

> **부서별 vs 용도별 — 왜 같은 유형에 두 가지 템플릿?** 뷰 기반이므로 둘 다 적용해도 된다. 부서별 뷰 + 용도별 뷰를 동시에 세팅하면 같은 상품을 "개발팀" 뷰와 "IT장비" 뷰에서 동시에 볼 수 있다. 온보딩에서는 하나만 선택하게 하되, 이후 AI가 "용도별로도 보면 편할 것 같아요"라고 제안 가능.

**E. 이커머스/유통형 + 부서별**
```
초기 뷰 세트 (필터: context.team)
├── 📌 경영지원 (관리자 스코프)
├── 📌 MD/상품기획
├── 📌 마케팅
├── 📌 물류/SCM
├── 📌 CS/운영
└── 📌 개발
예산 정책: 월 ₩5.3M (중앙값), 다양성 높음
승인: 팀장 ₩150K 이하 자동
```

**F. 전문서비스형 + 부서별**
```
초기 뷰 세트 (필터: context.team)
├── 📌 경영관리 (관리자 스코프)
├── 📌 영업/세일즈
├── 📌 컨설팅/서비스
└── 📌 지원/총무
예산 기준: 월 ₩1.7M (중앙값), 최소 규모
승인: 관리자 일괄 승인 (소규모 특성)
```

### 6-C. 엣지케이스 대응 매트릭스

| # | 엣지케이스 | 발생 상황 | AI 대응 | 폴백 |
|---|-----------|----------|---------|------|
| 1 | **1인 기업 / 프리랜서** | 직원 "1명", 부서 없음 | 뷰 설정 단계 완전 스킵. "나중에 팀이 생기면 추가할 수 있어요" | 용도형 2-3개 뷰 자동 생성 |
| 2 | **부서+지점 혼합** | 삼천리모터스처럼 본사 부서 + 전시장/서비스 센터 공존 | 복합 뷰 세트 제공: 지점별 뷰(`context.location`) + 부서별 뷰(`context.team`)를 동시 생성. 상품이 양쪽에 동시 노출됨 | 플랫 구조로 지점과 부서를 병렬 나열 |
| 3 | **조직명을 모르겠음** | 막 창업한 회사, 조직 미확정 | 용도형 뷰를 기본 추천 + "조직이 잡히면 부서별 뷰도 추가할 수 있어요" | "전체" 뷰만 + 자유롭게 추가 |
| 4 | **뷰 구조가 맞지 않게 됨** | 부서형으로 했는데 지점이 생기거나, 지점이 통폐합됨 | AI가 구매 패턴 변화 감지 → "지점별 뷰를 추가할까요?" / "이 뷰에 해당하는 구매가 없는데 정리할까요?" | 수동 뷰 관리 UI |
| 5 | **기존 에어서플라이 고객** | 이미 폴더/팀원/결제수단이 있음 | 기존 폴더 구조를 뷰로 자동 변환 + "이런 점을 개선하면 좋겠어요" 제안 | 기존 구조 1:1 뷰 변환 |
| 6 | **브랜드 복수 운영** | 도레도레처럼 도레도레/마호가니/아모르나폴리 등 멀티 브랜드 | 브랜드별 뷰(`context.brand`) + 매장별 뷰(`context.location`) 복합 생성 | 브랜드 구분 없이 매장 뷰만 |
| 7 | **뷰 수 폭발** | 스파크플러스(49개), 패스트파이브(48개) 수준 | 자동 그루핑 뷰 제안 — "서울 강남권", "서울 마포/홍대권" 등 상위 뷰 + 하위 필터 | 검색/필터 기능으로 보완 |
| 8 | **외국계 기업** | 뷰 이름이 영문, 글로벌 조직 구조 | 영문 뷰 지원 + HQ/Regional 구조 자동 감지 | 언어 무관하게 처리 |
| 9 | **프랜차이즈 가맹점** | 본사가 가맹점 구매를 관리하고 싶음 | 본사=Admin(전체 뷰 접근), 가맹점=Requester(자기 뷰만) + 가맹점별 예산 상한 | 개별 기업으로 분리 |
| 10 | **Layer 0에서 오래 머무는 유저** | 상품 탐색만 하고 회사 정보 입력 안 함 | 3회 이상 탐색 시 부드러운 넛지: "회사 정보 알려주시면 더 정확한 추천이 가능해요" | 강제하지 않음 — 탐색 경험 자체가 리텐션 |
| 11 | **Layer 2에서 이탈** | 결제수단 등록 거부 | "결제수단 없이도 구매요청서는 만들 수 있어요. 관리자가 나중에 결제수단을 연결하면 돼요." | 구매요청서 PDF 내보내기로 오프라인 대안 제공 |
| 12 | **관리자가 아닌 사람이 먼저 가입** | 구성원이 먼저 써보고, 관리자에게 권유하는 케이스 | "관리자를 초대하시면 승인/결제 기능을 활성화할 수 있어요" | 구성원만으로도 탐색+요청까지 가능 |

#### 특별 케이스: "관리자 vs 구성원" 진입 분기

Progressive Onboarding에서 중요한 분기점 — **누가 먼저 가입하느냐**에 따라 경험이 달라져야 한다.

```
[가입 시 1개 질문만]

🤖 "혹시 회사에서 구매를 관리하는 분이세요, 
    아니면 필요한 물건을 요청하는 분이세요?"

   [구매 관리자예요]  [물건 요청하는 사람이에요]
```

**관리자 진입 시:**
- "가이드 투어" 비중을 높이고, 세팅 완성도를 높이는 방향
- Layer 1→2→3→4를 비교적 빠르게 안내 (그래도 각 단계 [나중에] 가능)
- 와우 포인트: "이전에 반나절 걸리던 세팅이 5분에 끝남"

**구성원 진입 시:**
- Layer 0에서 바로 상품 탐색 → 가치 체험
- Layer 1까지만 자연스럽게 (회사명, 내 부서 정도)
- 구매요청까지는 가능하되, 결제/승인 설정은 "관리자 초대" 유도
- 와우 포인트: "말했더니 3초 만에 찾아줌"

---

**핵심:** 기존 고객은 Day 1부터 "우리 회사 과거 구매 대비" 비교가 가능 → 와우 모먼트가 즉시 발동.

## 7. 와우 모먼트 설계

### 와우 모먼트 공식

```
와우 = (기대보다 빠른 속도) × (기대보다 높은 품질) × (내 맥락에 맞는 결과)
```

### Day 1 와우 vs Month 2 와우: 분리 설계

기존 문제: "우리 회사 지난달 대비 18% 절감"은 데이터가 쌓여야 작동 → 신규 기업에게 무의미
해결: Day 1 와우는 데이터 없이 작동하는 것으로, Month 2 와우는 축적된 데이터로 설계

### 타임라인

| 시점 | 와우 모먼트 | 의존 데이터 | 역할 | 비교 기준 |
|------|-----------|-----------|------|---------|
| **온보딩** | "세팅이 대화로 15분에 끝남" | 없음 | 진입 장벽 해소 | — |
| **Day 1 첫 탐색** | "30분 걸리던 게 30초에 끝남" | 없음 (시장 데이터) | **획득/전환** | 카테고리 평균가, 유형별 인기 상품 |
| **Day 1 첫 구매요청** | "승인까지 자동으로 갔음" | 없음 (승인 플로우) | **획득/전환** | — |
| **Day 1 구매 직후** | "영수증이 실시간으로 잡힘" | 없음 (즉시 기록) | 정산 와우 예고 | — |
| **Week 2+** | "지난번 거 재주문해줘" → 즉시 | 최소 1건 이력 | 습관 형성 | — |
| **Month 2+** | "우리 회사 평균 대비 15% 절감" | 누적 구매 데이터 | **리텐션/Lock-in** | 자사 이력 |
| **Month 2+ 정산** | "원클릭 정산 — 90% 시간 단축" | 1개월 구매 축적 | **리텐션/Lock-in** | 기존 수작업 정산 |
| **Quarter 2+** | "이번 분기 절감액 ₩2.4M" | 분기 데이터 | 갱신 근거/업셀 | 자사 이력 + 유형 벤치마크 |

### Day 1 와우의 비교 맥락 전략 (콜드스타트 해결)

신규 기업에게 자사 이력이 없을 때 쓸 수 있는 비교 기준:

| 비교 기준 | 예시 | 데이터 소스 |
|----------|------|-----------|
| **카테고리 평균가** | "이 카테고리 평균 대비 12% 저렴" | 에어서플라이 전체 거래 데이터 |
| **유형별 인기 상품** | "IT 오피스형 기업들이 가장 많이 사는 제품" | 유형 분류 데이터 |
| **시장 데이터** | "최저가 기준 3개 쇼핑몰 비교" | 상품DB 크롤링 |
| **객관적 스펙** | "리뷰 4.7 / 배송 1.2일 / 반품률 0.3%" | 쇼핑몰 데이터 |

→ **자사 이력이 쌓이면 점진적으로 개인화 비교로 전환:**

```
Day 1:   "IT 오피스형 기업 평균 대비 12% 저렴"
Month 1: "우리 회사 지난달 구매 대비 8% 저렴"
Month 3: "우리 부서 분기 평균 대비 15% 절감, 전체 IT 오피스형 대비 상위 20%"
```

### 에어서플라이 기존 가치의 와우 전환

기존에 "한 달 뒤 정산 때 깨닫던 가치"를 Lambda에서는 실시간으로 느끼게 한다:

| 기존 가치 | 기존 체감 시점 | Lambda 체감 시점 | 어떻게 |
|----------|-------------|-----------------|--------|
| 구매처 통합 | 여러 번 써본 후 | **첫 구매 시** | "어디서 샀든 여기서 다 보여요" 메시지 |
| 세부내역 가시화 | 정산 시 | **구매 직후** | 실시간 지출 기록 즉시 노출 |
| 보안 (계정 통합) | 설정 후 | **온보딩 시** | "팀원들 개인 계정 공유 없이 구매 가능" 안내 |
| 정산 자동화 | 첫 정산 시 | **구매 축적 중** | "🧾 영수증 12건 자동 수집 완료" 실시간 노출 |
| 지점 관리 | 지점 추가 후 | **온보딩 시** | "지점별 뷰를 만들어드릴까요?" |

## 8. 과금 모델 전환의 전략적 의미

### As-Is → To-Be

```
As-Is (구매대행 수수료):
  기업이 느끼는 것: "물건값 + α(수수료)" = "구매할 때마다 돈이 더 나간다"
  회계 분류: 매입원가(COGS)에 포함 가능성 → 원가율 상승으로 보임
  심리: 비용 = 저항

To-Be (SaaS 구독 + 토큰 과금):
  기업이 느끼는 것: "월 구독료 + AI 사용량" = "업무 도구에 투자한다"
  회계 분류: 판관비(SG&A) / 복리후생비 → 매입원가와 완전 분리
  심리: 투자 = 합리성
```

### 세 가지 구조적 이점

**① 심리적 프레이밍: "비용" → "투자"**

수수료: "물건 살 때마다 돈이 나간다" (가치 인식 없이 비용만 보임)
SaaS+토큰: "AI가 나 대신 일했다" (토큰 소모 순간마다 가치 인식 동반)

Slack에 메시지 보낼 때마다 "돈 나간다"고 느끼지 않는 것과 같은 원리.

**② 회계적 분리: 매입원가 vs 판관비**

| 항목 | 수수료 모델 | SaaS 모델 |
|------|-----------|----------|
| 회계 분류 | 매입원가(COGS) | 판관비(SG&A) / 복리후생비 |
| 원가율 영향 | 원가율 상승으로 보임 | 원가율과 무관 |
| 예산 출처 | 구매 예산에서 차감 | IT/SaaS 예산에서 집행 |
| 승인 난이도 | "왜 물건값이 더 비싸?" | "팀 생산성 도구 도입" |

**③ 수익 예측 가능성**

| 지표 | 수수료 모델 | SaaS + 토큰 모델 |
|------|-----------|----------------|
| 매출 예측 | 구매 건수에 종속 (변동 큼) | 구독 = 고정 + 토큰 = 사용량 비례 |
| 이탈 신호 | 구매 급감 시 이미 늦음 | 토큰 사용량 감소로 선제 대응 |
| 업셀 경로 | 통제 불가 | 플랜 업그레이드 + 토큰 팩 |

### 과금 구조 초안

```
┌─── Free Tier ──────────────────────────────┐
│  월 50회 에이전트 실행                        │
│  상품 탐색 + 비교만 가능                      │
│  구매요청서 수동 작성                         │
│  → 와우 모먼트 체험용                        │
└────────────────────────────────────────────┘

┌─── Starter (₩99,000/월) ───────────────────┐
│  월 500 토큰 포함                            │
│  상품 탐색 + 비교 + 자동 구매요청              │
│  기본 승인 플로우                             │
│  구성원 5명까지                               │
│  → 소규모 팀/부서 단위                        │
└────────────────────────────────────────────┘

┌─── Business (₩299,000/월) ─────────────────┐
│  월 2,000 토큰 포함                          │
│  전체 기능 + 관리자 대시보드                   │
│  지출 분석 + 구매 건강도                       │

│  → 기업 전체 도입                             │
└────────────────────────────────────────────┘

토큰 소모 기준 (예시):
  상품 탐색 1회           = 1 토큰
  상품 비교/분석 1회        = 2 토큰
  구매요청서 자동 작성      = 3 토큰
  지출 분석 리포트         = 5 토큰
  커넥터 호출 (향후)       = 커넥터별 상이
```

## 9. 구성원 vs 관리자 화면 분리 수준

### 결정: 같은 앱, 역할 기반 점진적 확장 (Option A+)

#### 검토한 선택지

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **A. 역할 기반 뷰 전환** | 하나의 앱에서 역할에 따라 메뉴/기능이 달라짐 | 개발 공수 최소, 관리자도 직접 구매 가능, Progressive Onboarding과 자연 연결 | 관리자 대시보드 공간 제약, 멀티태스킹 어려움 |
| B. 별도 "관리 모드" | 같은 앱이지만 관리 모드 전환 시 레이아웃 변경 | 관리자에게 충분한 공간, 구성원 화면 단순함 유지 | 사실상 두 앱 설계, "지금 어떤 모드?" 혼란 |
| C. 완전 별도 앱 | 구성원용 + 관리자용 별도 앱 | 각각 UX 최적화 가능 | 개발 2배, 앱 전환 필요, MVP에 과한 투자 |

#### 왜 Option A인가

1. **Lambda의 관리자 = 구매도 하는 사람.** 에어서플라이 실데이터에서 관리자(경영지원/재무)도 직접 구매하는 비율이 높다. 별도 앱이면 "구매하려면 다른 앱 열어야 해"가 됨.
2. **Progressive Onboarding 철학과 일관.** Layer 0~4로 기능이 점진적으로 열리는 구조에서, 관리 기능도 "역할이 부여되면 아이콘이 나타나는" 방식이 자연스럽다.
3. **MVP 10명 이하 팀에 적합.** 두 개 앱을 만들 여유가 없다.
4. **관리자 대시보드 공간 문제는 메인 영역으로 해결.** 사이드바에 대시보드를 넣는 게 아니라, 📊 아이콘 클릭 시 메인 영역 전체가 대시보드로 바뀜.

```
  🛒 스토어 — 에어서플라이 상품 카탈로그
```

#### 핵심 설계 원칙

**① 구분선(────)이 경계선**

Icon Rail에서 구분선 위 = 모든 역할 공통(대화, 내 상품, 스토어). 구분선 아래 = 역할에 따라 점진적으로 추가. 관리자가 아닌 사람에게는 구분선 아래가 아예 보이지 않음. "기능이 숨겨져 있다"는 느낌이 아니라 "내 역할에 맞는 깔끔한 화면"으로 인식.

**② 관리 아이콘 클릭 = 메인 영역 전체 전환**

관리자가 📊을 클릭하면 메인 영역이 대시보드로 바뀜. 이때 사이드바 컨텍스트 패널은 해당 모드에 맞는 필터/상세를 표시. 💬을 다시 클릭하면 대화로 복귀 — 모드 전환이 아니라 **탭 전환**에 가까운 인지 모델.

**③ 대화 안에서도 관리 기능 접근 가능**

관리자가 대화 중 "이번 달 지출 현황 보여줘"라고 말하면 AI가 대시보드 데이터를 대화 안에서 인라인으로 보여줌. 반드시 📊 탭으로 이동할 필요 없음. 대화가 메인이라는 원칙 유지.

**④ 알림 뱃지는 역할에 따라 다르게 작동**

| 역할 | 💬 뱃지 | ✅ 뱃지 | 📊 뱃지 |
|------|---------|---------|---------|
| Requester | 내 요청 상태 변경 (승인됨/반려됨/배송 완료) | — | — |
| Buyer | 동일 | 내가 승인할 건 N개 | — |
| Admin | 동일 | 전체 승인 대기 N개 | 예산 초과 알림, 이상 지출 감지 |

#### Progressive Onboarding과의 연결

```
Layer 0: Icon Rail = 💬만 활성화 (다른 아이콘은 흐리게 + "상품을 탐색하면 여기서 볼 수 있어요" 툴팁)
Layer 1: 📦 내 상품 활성화 (유형 진단 → 뷰 생성 후)
Layer 2: 🛒 스토어 완전 활성화 + 결제 가능
Layer 3: 역할에 따라 구분선 아래 아이콘 등장
Layer 4: 📊 대시보드 + 🔒 정책 활성화 (데이터 축적 후)
```

→ 처음엔 채팅 하나만 보이다가, 사용할수록 아이콘이 하나씩 나타나는 경험. **게임의 기능 해금(unlock)과 동일한 심리 메커니즘.**

## 10. 승인 플로우 상세 설계

### 권한 모델: RBAC + 뷰 스코프 + 예산 정책

#### 세 가지 기본 역할

| 역할 | 할 수 있는 것 | 할 수 없는 것 | 전형적 직책 |
|------|-------------|-------------|-----------|
| **Requester** | 상품 탐색, 구매요청 생성, 본인 요청 상태 확인 | 직접 결제, 타인 요청 승인, 정책 변경 | 일반 구성원 |
| **Buyer** | Requester 전체 + 한도 내 자동결제 + 지정된 뷰의 요청 승인 | 정책 변경, 멤버 관리, 결제수단 관리 | 팀장, 부서 리드 |
| **Admin** | 전체 기능 + 정책 설정 + 멤버 관리 + 결제수단 + 대시보드 | — | 경영지원팀장, 재무, 대표 |

#### 역할 × 뷰 스코프 매트릭스

역할만으로는 부족하다. "마케팅팀 팀장이 개발팀 구매를 승인하면 안 된다"는 제약이 필요. **뷰 스코프**가 이 문제를 해결한다.

```
김팀장 (Buyer, 스코프: [마케팅팀 뷰])
├── 마케팅팀 뷰 내 구매요청 → 승인 가능
├── 개발팀 뷰 내 구매요청 → 승인 불가 (보이지도 않음)
└── 본인 구매 → 한도 내 자동결제

이관리자 (Admin, 스코프: [전체])
├── 모든 뷰의 구매요청 → 승인 가능
├── Buyer가 승인한 건 중 한도 초과 → 2차 승인
└── 정책/멤버/결제수단 관리
```

#### 예산 정책 구조

```
예산 정책 = 역할 × 뷰(선택) × 조건

예시:
┌───────────────────────────────────────────────────────┐
│  정책 이름: "일반 구매 승인"                              │
│  적용 대상: Requester (전체)                             │
│  조건:                                                 │
│    건당 ₩50,000 이하 → Buyer 승인                       │
│    건당 ₩50,000~₩300,000 → Buyer 승인 + Admin 통보     │
│    건당 ₩300,000 초과 → Admin 직접 승인                  │
│    월 누적 ₩500,000 초과 → Admin 직접 승인               │
│  예외: "긴급 구매" 플래그 시 → Admin에게 즉시 알림         │
└───────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────┐
│  정책 이름: "IT장비 특별 승인"                            │
│  적용 대상: 모든 역할                                    │
│  적용 뷰: IT장비/가전 뷰                                │
│  조건:                                                 │
│    건당 무관 → 반드시 Admin 승인                          │
│  이유: IT 자산은 자산 관리 대장에 등록 필요                  │
└───────────────────────────────────────────────────────┘
```

→ 유형별 세팅 템플릿(6-B)에서 기본 정책이 자동 생성되고, Admin이 세부 조정.

#### 핵심 UX 패턴: 대화가 끊기지 않는다

**패턴 1: 투명한 자동결제 (Buyer/Admin 본인 구매)**

```
👤 "이걸로 해줘"
🤖 "결제 완료! ₩76,500
    배송 예정: 4/10 (목)
    🧾 지출 기록에 자동 반영됐어요."
```

승인 과정이 전혀 보이지 않음. Buyer/Admin이 본인 한도 내에서 구매할 때.

**패턴 2: 대화 내 즉석 승인 요청 (Requester → Buyer)**

```
👤 "이걸로 해줘"
🤖 "₩76,500은 직접 결제 한도(₩50,000)를 넘어요.
    김팀장님에게 승인 요청을 보낼까요?"
    
    ┌─────────────────────────────────────┐
    │  📋 구매요청서 (자동 생성)             │
    │  상품: JBL Flip 6 블루투스 스피커      │
    │  금액: ₩76,500                       │
    │  용도: 회의실 비품 (대화에서 자동 추출)   │
    │  추천 사유: 카테고리 평균 대비 15% 절감  │
    │                                      │
    │  [승인 요청 보내기]  [취소]             │
    └─────────────────────────────────────┘

👤 [승인 요청 보내기]

🤖 "김팀장님에게 슬랙으로 알림을 보냈어요.
    승인되면 바로 알려드릴게요.
    
    다른 필요한 거 있으세요? 기다리는 동안 찾아볼게요."
```

핵심: **승인 대기 중에도 대화가 이어진다.** 사용자를 블로킹하지 않음.

**패턴 3: 승인자 측 경험**

```
[승인자의 Lambda — ✅ 승인 대기 탭, 또는 슬랙 알림]

🔔 구매요청이 도착했습니다.

   ┌─────────────────────────────────────────────────┐
   │  요청자: 박지현 (마케팅팀)                          │
   │  상품: JBL Flip 6 블루투스 스피커 — ₩76,500        │
   │  용도: 회의실 비품                                 │
   │                                                  │
   │  💡 AI 판단 근거:                                 │
   │  • 카테고리(스피커) 평균가 ₩89,000 대비 14% 저렴    │
   │  • 리뷰 4.7 / 배송 내일 도착                       │
   │  • 마케팅팀 이번 달 잔여 예산: ₩423,500             │
   │                                                  │
   │  [승인]  [반려 + 사유 입력]  [대화 보기]             │
   └─────────────────────────────────────────────────┘
```

핵심: **AI 판단 근거가 승인 화면에서 기본 노출.** 승인자가 별도로 가격 비교하거나 예산 확인할 필요 없음. "승인 버튼 하나만 누르면 되는" 경험.

**패턴 4: 승인 결과 비동기 알림**

```
[몇 분~몇 시간 후, 요청자의 대화에 알림 삽입]

🔔 승인 완료!
🤖 "김팀장님이 방금 승인했어요.
    결제를 진행할게요... 완료! ₩76,500
    배송 예정: 4/10 (목)"
```

### 예외 처리

| # | 예외 상황 | 시스템 행동 | UX |
|---|----------|-----------|-----|
| 1 | **승인자 부재 (24h 무응답)** | 에스컬레이션: 같은 뷰 스코프의 다른 Buyer → 없으면 Admin에게 자동 전달 | 요청자에게 "아직 대기 중이에요. 다른 승인자에게 전달할까요?" |
| 2 | **반려** | 반려 사유 + 승인자 메모를 요청자 대화에 전달 | "반려됐어요. 사유: [X]. 수정해서 다시 요청할까요?" + [재요청] 버튼 |
| 3 | **긴급 구매** | 요청자가 "긴급" 플래그 설정 → 승인자에게 즉시 푸시알림 + 알림 주기 단축(1h) | "긴급으로 표시할게요. 김팀장님에게 바로 알림이 갑니다." |
| 4 | **승인자 본인이 요청** | Buyer가 본인 한도 초과 구매 시 → 상위 Admin에게 승인 요청 | "₩300,000은 본인 승인 한도를 넘어요. 이관리자님에게 승인 요청을 보낼까요?" |
| 5 | **승인 후 가격 변동** | 결제 시점에 가격이 승인 당시보다 상승 → 재승인 요청 (하락은 그대로 진행) | "승인 당시 ₩76,500이었는데 지금 ₩82,000이에요. 재승인이 필요해요." |
| 6 | **결제수단 미등록 상태에서 승인** | 승인은 완료하되, 결제 실행 보류. Admin에게 결제수단 등록 요청. | "승인됐어요! 결제수단이 아직 없어서 관리자에게 등록 요청을 보냈어요." |
| 7 | **동일 상품 반복 요청** | AI가 패턴 감지 → "이 상품은 매달 구매하시네요. 자동 재주문을 설정할까요?" + 승인 규칙에 "반복 구매 자동승인" 추가 가능 | 관리자가 정책에서 "반복 구매 자동승인 품목" 지정 가능 |
| 8 | **예산 소진** | 뷰(팀/부서)의 월 예산이 초과된 상태에서 요청 → Admin에게 예산 추가 승인 또는 다음 달 이월 선택 | "마케팅팀 이번 달 예산이 초과됐어요. 관리자에게 추가 예산을 요청할까요, 다음 달에 구매할까요?" |

### 알림 체계

| 이벤트 | 대상 | 채널 (우선순위) | 즉시 vs 배치 |
|--------|------|----------------|-------------|
| 승인 요청 도착 | 승인자 | ① Lambda 앱 내 ✅ 뱃지 ② 슬랙 DM ③ 이메일 | 즉시 |
| 긴급 승인 요청 | 승인자 | ① 슬랙 DM + 멘션 ② 앱 푸시 ③ 이메일 | 즉시 + 1h 리마인더 |
| 승인/반려 결과 | 요청자 | ① Lambda 대화 내 인라인 ② 슬랙 DM | 즉시 |
| 결제 완료 + 배송 시작 | 요청자 + Admin | ① Lambda 대화 ② 슬랙 (Admin은 배치) | 즉시(요청자) / 일일 요약(Admin) |
| 24h 무응답 에스컬레이션 | 차순위 승인자 | ① 슬랙 DM ② 이메일 | 자동 |
| 예산 80% 도달 | Admin | ① Lambda 📊 뱃지 ② 슬랙 | 배치 (일일) |
| 예산 100% 초과 | Admin + 해당 뷰 Buyer | ① Lambda + 슬랙 | 즉시 |

> **MVP 범위 제한:** 슬랙 연동을 1순위로 구현. 이메일은 슬랙 미연동 기업용 폴백. 앱 푸시는 모바일 이후.

```
   ┌─────────────────────────────────────────────────┐
   │  💡 추천 승인 규칙                                │
   │                                                  │
   │  일반 구성원 구매:                                 │
   │    ₩50,000 이하 → 팀장 자동승인                    │
   │    ₩50,000 초과 → 팀장 확인 후 승인                │
   │    ₩300,000 초과 → 관리자 직접 승인                │
   │                                                  │
   │  팀장 본인 구매:                                   │
   │    ₩300,000 이하 → 자동결제                        │
   │    ₩300,000 초과 → 관리자 승인                     │

```

→ 유형별 세팅 템플릿(6-B)의 승인 기준선이 여기서 기본값으로 사용됨.
